## Порівняння підходів до побудови RAG: Naive, Sentence-Window та Parent-Child Retrieval

Цей звіт порівнює три поширені підходи до реалізації Retrieval Augmented Generation (RAG) систем: Naive RAG, Sentence-Window Retrieval та Parent-Child Retrieval (також відомий як Auto-Merging Retrieval). Кожен з цих підходів має свої переваги та недоліки, які впливають на точність, релевантність та повноту відповідей, що генеруються великими мовними моделями (LLM).

### 1. Naive RAG (Базовий пошук за індексом)

**Механізм:**
У підході Naive RAG документи розбиваються на фрагменти (chunks) фіксованого розміру. Ці фрагменти індексуються, і під час пошуку система знаходить найбільш схожі фрагменти до запиту користувача. Знайдені фрагменти передаються LLM як контекст для генерації відповіді.

**Переваги:**
*   **Простота реалізації:** Це найпростіший підхід для початку роботи з RAG.

**Недоліки:**
*   **Втрата точності контексту:** Фрагменти фіксованого розміру часто містять непов'язаний контент, що може заплутати LLM і призвести до менш точних відповідей.
*   **Інформаційне перевантаження:** Великі фрагменти можуть містити зайву інформацію, ускладнюючи для LLM зосередження на дійсно важливих деталях.
*   **Відсутність деталізації:** Важливі деталі можуть бути загублені у великих текстових блоках, знижуючи точність.
*   Не гарантує точних відповідей, оскільки релевантний контекст може бути розкиданий або розбавлений.

### 2. Sentence-Window Retrieval

**Механізм:**
Sentence-Window Retrieval покращує традиційний пошук, зосереджуючись на окремих реченнях та їхньому навколишньому контексті.
1.  Документи розбиваються на окремі речення (або дуже малі фрагменти). Ці малі фрагменти індексуються для пошуку.
2.  Коли знайдено релевантний малий фрагмент (речення), система також витягує "вікно" навколишніх речень (сусідів), щоб надати додатковий контекст.
3.  Ці витягнуті малі фрагменти та їхні сусіди потім конкатенуються, утворюючи більший контекст для LLM.

**Переваги:**
*   **Висока деталізація контексту:** Покращує точність, виконуючи пошук за меншими, більш сфокусованими одиницями.
*   **Зменшення шуму:** Зменшує кількість нерелевантної інформації під час пошуку.
*   **Проста конкатенація:** При роботі на рівні речень без перекриттів конкатенація стає тривіальною.

**Недоліки:**
*   **Обробка меж вікна:** Потребує ретельного контролю меж вікна (початок і кінець документа), щоб не вийти за їхні межі.
*   **Метадані:** Необхідно відстежувати метадані, такі як вихідний документ та індекс фрагмента, для коректного вилучення сусідніх фрагментів.

### 3. Parent-Child Retrieval (Auto-Merging Retrieval)

**Механізм:**
Parent-Child Retrieval (або Auto-Merging Retrieval) використовує ієрархічну структуру для організації даних.
1.  Документи спочатку розбиваються на великі сегменти, які називаються "батьківськими фрагментами" (parent chunks).
2.  Ці батьківські фрагменти потім поділяються на менші сегменти, відомі як "дочірні фрагменти" (child chunks).
3.  Дочірні фрагменти пов'язані зі своїми відповідними батьківськими фрагментами.
4.  Пошук виконується за меншими дочірніми фрагментами.
5.  Якщо знайдено релевантні дочірні фрагменти, їхні більші батьківські фрагменти (або об'єднаний контекст з кількох релевантних дочірніх фрагментів та їхнього батьківського) надаються LLM. Це дозволяє досягти точного пошуку (з дочірніх фрагментів) та багатого контексту (з батьківських).

**Переваги:**
*   **Поєднання переваг:** Об'єднує переваги менших фрагментів (точність) та більших фрагментів (всебічний контекст).
*   **Збагачення за запитом:** Надає додатковий контекст лише тоді, коли це необхідно.
*   **Зменшення хибних спрацьовувань:** Зменшує кількість хибних спрацьовувань, виконуючи пошук за меншими одиницями.

**Недоліки:**
*   **Складність реалізації:** Більш складна логіка індексації та пошуку через ієрархічну структуру.
*   **Дизайн відносин:** Потребує ретельного проектування відносин між батьківськими та дочірніми фрагментами.

### Висновок

Вибір підходу RAG залежить від конкретних вимог до системи, доступних ресурсів та бажаного рівня точності та повноти. 

*   **Naive RAG** є хорошою відправною точкою для простих випадків використання, де точність не є критичною.
*   **Sentence-Window Retrieval** пропонує значне покращення точності, особливо для доменів, що вимагають високої точності, таких як юридичні або медичні тексти, завдяки фокусу на гранулярному контексті.
*   **Parent-Child Retrieval** (Auto-Merging Retrieval) є найбільш складним, але й найпотужнішим підходом, який забезпечує найкращий баланс між точністю та повнотою контексту, дозволяючи LLM генерувати високоякісні та релевантні відповіді. Він ідеально підходить для складних сценаріїв, де потрібен глибокий та всебічний контекст.

Для більшості реальних застосувань, що вимагають високої продуктивності RAG, рекомендується розглядати Sentence-Window або Parent-Child Retrieval, незважаючи на їхню більшу складність реалізації.
